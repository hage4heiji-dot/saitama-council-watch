# ADR-0027: 議員任期履歴(市議会議員選挙結果の取り込みとマトリクス表示)

## ステータス
一部実装(PR1: 2003年のみ)。承認済み(2026-07-23)

## コンテキスト
ユーザーから「議員を過去30年間までさかのぼって任期がわかるようにしてほしい。横軸年、縦軸人のマトリクスで表現してほしい」との要望があった。

事前調査の結果、2点の重要な制約が判明した。

1. **「30年」は物理的に不可能**。さいたま市自体が2001年5月(浦和・大宮・与野の合併、2005年に岩槻編入)に成立した市であり、市議会の統一地方選挙データは合併後最初の選挙である2003年(平成15年)4月13日執行分より前が存在しない。現実的な上限は約23年(2003年〜現在)である。
2. 過去の選挙結果PDF(`https://www.city.saitama.lg.jp/006/009/kakonosenkyokekka/`配下)は、`pdf-parse`(このリポジトリの既存PDF抽出基盤、内蔵pdfjsが古い)ではフォントのグリフを解決できず本文が完全に空文字列になることを実機検証で確認した。一方、poppler-utilsの`pdftotext -layout`は同じPDFを正しく解読できる。2003年〜2023年の全6回の統一地方選挙PDFで実際にダウンロード・抽出検証済み。

「捏造しない」という原則(docs/design/00-constitution.md)を踏まえ、実在する範囲のデータのみを扱う。

## 決定

### 1. PDFテキスト抽出はpdftotext(poppler-utils)を新たに使う
`apps/api/src/infrastructure/documentText/extractPdfTextViaPoppler.ts`を追加し、`apps/api/Dockerfile`のruntimeステージに`poppler-utils`を追加した。既存の`pdf-parse`ベースの抽出(`extractPdfText.ts`、`extractPositionedPdfText.ts`)は他の機能(予算・議案審議結果等)で正常に動作しているため変更しない。このPDF種別(市発行、eDocument Library製)に限定した対処とする。

### 2. `CouncilTerm`という新モデルで、選挙結果を単独保持する
過去の当選者は既存`Legislator`と確実に一致するとは限らない(引退・表記揺れ)ため、`CouncilTerm`はスクレイピング結果(氏名・選出区・党派・得票数)を自身で保持し、`legislatorId`は氏名の完全一致(空白除去後)で確信を持って一致できた場合のみ設定する。これはADR-0023の`CommitteeMeeting.meetingId`のnullable方針と同じ考え方である。

### 3. 任期日付はEXPLICIT/ASSUMEDを区別する
2003年PDFには「当選人の任期 平成15年５月１日〜平成19年４月30日」という明記があるが、他の年度(2007〜2023)にはこの表記がない。明記がある場合はそのまま採用しEXPLICITとする。ない場合は、2003年の実データ(投票翌月1日開始・4年後の前日満了)から一般化した「統一地方選挙の慣例」でASSUMEDとして算出する。**この慣例は2003年の1件のみから一般化した仮定であり、地方自治法上の任期起算日の一般原則を確認できていない。** 将来、他の年度に明記があるPDFが見つかった場合はEXPLICITで上書きし、この仮定との整合性を検証すること。

### 4. 任期途中の辞職・繰上当選は2行に分けて記録する
2003年の実データで、辞職→繰上補充選挙という事例が2件(西区・中央区)見つかった。原本の脚注(自由文)から`resignedName`・`resignedDate`・`successionDate`・`successorName`を抽出し、辞職した議員の行に`resignedDate`・`successorRawName`を追加、繰上当選した人物には`origin=runner_up_succession`の新しい行を追加する(`predecessorRawName`で対応関係を保持)。脚注の文面が既知の定型文と一致しない場合は解決しない(捏造しない)。

### 5. 一度きりのバックフィルジョブとし、cronには登録しない
過去の確定した選挙結果は将来変化しないため、`npm run --workspace apps/api ingest:council-terms`で手動実行する一度きりのジョブとする。2027年の次回統一地方選挙以降は、`apps/api/src/batch/jobs/ingestCouncilTerms.ts`の`ELECTIONS`配列に新しい選挙を追記して再実行する運用を想定する。

### 6. PDFのURLではなく、年度見出しから辿ってPDFを解決する
索引ページ(例: `p018209.html`)は複数選挙をまとめて掲載しており、リンクの表示テキストには年度が含まれない(常に「得票数及び当選人（PDF形式：◯◯KB）」)。実際のHTML構造を確認した結果、`<h2>平成15年4月13日執行</h2>`という見出しの直後に対象リンクが続くことを確認したため、見出しテキストから辿って解決する(`saitamaElectionResultsScraper.ts`)。PDFのファイル名(`p018209_d`配下)はCMS内部の命名でいつ変わるか分からないため、URLを直接ハードコードしない。

## 進め方(複数PRに分割)
- **PR1(このコミット)**: スキーマ+パーサー+2003年分のみでエンドツーエンド実装。回帰テスト12件(`termBoundaryCalculation.test.ts`、`electionResultTableParsing.test.ts`、実データフィクスチャ使用)。
- **PR2(以降)**: 残り5回の選挙(2007/2011/2015/2019/2023)+既知の補欠選挙(2013年見沼区)を追加。年度ごとのPDF書式差異(任期明記の有無等)への対応。
- **PR3**: APIエンドポイント(`GET /api/v1/council-terms`)。
- **PR4**: フロントエンドのマトリクスUI(`/legislators/history`、CSS Gridベース、チャートライブラリ不使用。ADR-0023§3と同じ「質素なスタイル」方針を踏襲)。

## 結果(Consequences)

**メリット**
- 2003年の実データ(9区、当選43名+繰上当選2名)をpdftotextで正しく抽出できることを実機検証済み(パーサーの回帰テスト12件が実データで全て通過)。
- 辞職・繰上当選という任期途中の異動も、原本の脚注から実データとして捕捉できた。

**デメリット・今後の課題**
- ASSUMEDな任期日付算出ルールは2003年の1件からの一般化であり、他年度で反例が見つかる可能性がある。
- 2007年以降のPDFには任期明記がないため、それらの年度のCouncilTerm行はtermStartDate/termEndDateがASSUMEDになる(PR2で対応)。
- 落選者はCouncilTermとして保持しない(議席を得ていないため任期という概念に合わない)。将来「候補者としての立候補履歴」を扱う場合は別モデルが必要になる。
