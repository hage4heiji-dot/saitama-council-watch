# ADR-0028: 歳入予算の款別内訳可視化(「お金の流れ」v2)

## ステータス
承認済み(2026-07-24)

## コンテキスト
ADR-0024(歳出/支出の款別内訳、v1)は歳入(収入)側の対応を将来拡張として明示的に先送りしていた。今回、その歳入内訳を追加する。

歳出予算議案のPDFには「第１表 歳入歳出予算(補正)」という同一の表に歳入(款/項)と歳出(款/項)の両方が含まれており、ADR-0024で実装したパーサー(座標付きテキスト抽出+ヘッダー行ベースの列境界検出)は見出し文字列(「歳出」/「歳出合計」)と対象範囲を差し替えるだけで歳入側にも転用できる想定だった。実装前に、既存4fixture(当初予算2件・補正予算2件)に対して同ロジックを歳入版に読み替えて実データ検証した。

## 決定

### 1. スコープは歳出と同一: 普通会計(一般会計+特別会計)のみ。公営企業会計は引き続き対象外
公営企業会計(水道・下水道・病院事業会計)は「収益的収入及び支出」という全く別形式であり、ADR-0024と同様に対象外とする(別途将来ADR)。

### 2. 実データ検証で確認した事実
- 4fixture全件に`歳入`見出し・`歳入合計`終端行が存在し、ADR-0024が対応した表記揺れ(`　歳　入`/`歳　入`)に加え、半角スペース1つの変種(`　歳入`, `kokuhoInitialBudgetR8019.json`)も存在するが、JSの`\s`は全角スペース(U+3000)を含むため`stripAllWhitespace`の既存正規表現でそのまま対応できる(コード変更不要)。
- **均衡チェック**: 当初予算2件はいずれも`歳入合計 == 歳出合計`が金額単位で完全一致(一般会計: 716,000,000,000円、国保: 106,314,000,000円)。これは歳入歳出予算が均衡するという制度的性質に基づく独立した正しさの裏付けであり、歳入側パーサーが歳出側と同じロジックで正しく解析できていることを実データで確認した。
- 補正予算2件は歳入側・歳出側でカテゴリ数・合計額が異なる(例: 補正予算(第108号)は歳入4カテゴリ/274億円、歳出6カテゴリ/609億円)。これは補正予算表が変更のあった款のみを列挙するためで、両側で同じ款が変更されるとは限らない(ADR-0024がすでに文書化済みの性質)ため想定通り。
- ADR-0024が対応した「款が1つしかない小規模テーブル」の列検出エッジケース(`generalAccountAmendment2BudgetR8138.json`)は、歳入側では款が**3つ**しかないケースとしてさらに厳しい形で現れるが、ヘッダー行ベースの列境界検出ロジックはそのまま正しく機能する。新規fixtureの追加は不要。

### 3. 隠れた前提(バグではないが明示する)
`itemsAfterHeading`(旧`itemsAfterExpenditureHeading`)は見出し以降の全アイテムを無制限に返す。歳出パースでは歳出表が文書内最後の表なので無害だったが、歳入パースを汎用化すると「歳入見出し以降の残り」に後続の歳出表も含まれるようになる。現状は次の2点で偶然動作している:
- `findHeaderRow`が文書順で最初に一致したヘッダー行を返すため、歳入表自身のヘッダーが先に見つかる。
- 同一議案内の歳入表と歳出表は常に同じ列数(1列=当初予算/3列=補正予算)であるため、範囲が歳出表まで及んでも列検出結果は変わらない。

過剰設計(両端を厳密にスライスする)はせず、歳入解析結果に歳出カテゴリ名が混入していないことを検証する回帰テストと、パーサーのコード上の注意コメントで対応する。

### 4. 歳出/歳入は方向ごとに独立集計する
`aggregateBudget`(旧`aggregateExpenditureBudget`)は方向非依存のまま維持し、歳出用・歳入用でそれぞれ独立に呼び出す。取り込み時の「解析結果が空なら書き込まない(捏造しない)」判定は歳出・歳入を独立に評価する(片方の表だけ解析に失敗してももう片方は取り込む)。

### 5. `Budget`に`budgetType`(Prisma enum)を追加する
既存スキーマの他の閉じた集合(`MeetingType`等)はすべてPrisma enumで統一されており、この方針を踏襲する。一意キーを`(fiscalYear, accountName, category, budgetType)`に拡張する。`ingest-expenditure-budget`ジョブは既に本番稼働中でデータが存在するため、マイグレーションはADR-0024時点の2マイグレーション(テーブルが空だった前提でデフォルト値なしの列追加)とは異なり、`DEFAULT 'EXPENDITURE'`付きでバックフィルする必要がある。

### 6. 取り込みジョブ・ユースケースを`ingestBudget`にリネームする
`ingestExpenditureBudget`は歳入も扱うようになるため、命名の誠実性のためフルリネームする。`jobName`はユニーク制約もFKもない自由文字列の監査ログであることを確認済みで、リネームによる後方互換の懸念はない。

### 7. 表示は`/budget`ページに歳出/歳入切替ピルを追加する形とする
既存の会計年度・会計名ピルと同じUIパターンを踏襲し、`type`クエリパラメータ(省略時は歳出)で切り替える。既存のブックマークURLとの互換性を保つため、歳出をデフォルトとして`type`パラメータを省略可能にする。

## 結果(Consequences)

**メリット**
- 歳出側で確立した座標ベースのパーサー・集計ロジックをほぼそのまま再利用でき、新規スクレイパーもfixtureも不要。
- 歳入歳出の均衡チェックにより、実装の正しさを独立した角度から検証できた。

**デメリット・今後の課題**
- 公営企業会計は引き続きスコープ外。
- パーサーの「見出し以降無制限スキャン」という前提は、将来的に歳入表と歳出表の列数が食い違う議案が現れた場合に破綻しうる(回帰テストで検知する設計とし、いまは対応しない)。
