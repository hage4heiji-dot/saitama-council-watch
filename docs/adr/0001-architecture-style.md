# ADR-0001: モジュラーモノリス + ヘキサゴナルアーキテクチャの採用

## ステータス
承認済み(2026-07-06)

## コンテキスト
「さいたま市議会ウォッチ」は行政データをAIで整理・可視化するプラットフォームであり、以下の制約がある。

- 開発体制はソロ〜小規模を想定(想定規模: [[0005-scale-assumption]])
- インフラは単一VPS(Ubuntu 24.04)+ Docker([[0003-hosting-single-vps-docker]])
- 5年以上の運用を前提とし、AIモデル・DB・UIの変更が将来必ず発生する
- ドメイン(議員/会派/会議/議案/条例/投票/予算)は相互に密接に関連しており、独立してデプロイする必然性が薄い

## 検討した選択肢

1. **マイクロサービス**: ドメインごとにサービス分割。独立デプロイ・スケールが可能だが、分散トランザクション・サービス間通信・運用監視の複雑さが、単一VPS・小規模開発の実情に見合わない。
2. **素朴なモノリス(層分離なし)**: 実装は最速だが、AI/DB/スクレイパーの実装詳細がドメインロジックに漏れ出し、将来の変更(モデル差し替え等)の影響範囲が読めなくなる。
3. **モジュラーモノリス + ヘキサゴナル(採用)**: 単一デプロイ単位としての運用簡易性を保ちつつ、ドメイン層をDB/AI/スクレイパーの実装から切り離す。

## 決定
モジュラーモノリスを採用し、内部を以下の層に分離する。

- Domain層: Entity、Value Object、Repositoryインターフェース(ポート)
- Application層: ユースケース/サービス
- Infrastructure層: Prisma実装、Claude APIクライアント、スクレイパー、ファイルストレージ(アダプタ)

DB・AI・スクレイパーへの依存はすべてポート経由とし、実装はアダプタとして差し替え可能にする。

## 結果(Consequences)

**メリット**
- デプロイ・デバッグ・トランザクション管理が単純
- 将来的にAI処理のみを別サービスに切り出す「戦略的モノリス解体」が低コストで可能
- 「AIモデル変更/DB変更/UI変更の影響を最小化する」という保守方針([[../design/00-constitution.md]])を技術的に担保

**デメリット**
- AIパイプラインの負荷が急増した場合、Web/APIと同一ホストでリソース競合が起きうる
  → [[0008-worker-container-separation]] で緩和
- 層分離のための抽象化コストが小規模の初期実装に対しては若干過剰に見える

**将来の拡張性**
- Postgres→他DB、Claude→他モデル、単一VPS→クラウドのいずれも、該当アダプタの差し替えのみで対応可能
