# ADR-0012: Phase2(公開Webサイト最小版)における設計判断

## ステータス
承認済み(2026-07-06)

## コンテキスト
Phase2で議員一覧・会議一覧・議案一覧・全文検索の公開ページを実装するにあたり、実装中に確定した設計判断をまとめる。

## 決定事項

### 1. 議員データの取得元は「50音別」ページ
議員名簿には複数のページ(`/gikai/001/meibo/`の一覧表、`/gikai/001/002/001/`の50音別一覧)が存在するが、後者はふりがな(ルビ)・氏名・所属会派・プロフィールURL・写真を1ページに掲載しており、`Legislator.nameKana`(NOT NULL)を捏造せず取得できる。前者(名簿表)にはふりがなが含まれないため採用しなかった。

### 2. 会派名の重複疑義は原本サイトの「会派別」ページで検証済み
検索エンジンの要約で「自民党系の会派が2つ(9人+9人)存在する」という情報が示されたが、公式サイトの「会派別」一覧ページ(`/gikai/001/002/004/`)を直接確認した結果、「自民党市議団」は18人の単一会派として掲載されていることを確認した。二次情報(検索結果の要約)より一次情報(公式サイトの実データ)を優先する。

### 3. LegislatorFactionHistory.validFromは「観測日」であり「加入日」ではない
議員の所属会派スクレイピングはスナップショットであり、実際にいつその会派に加入したかは原本から取得できない。`validFrom`にはスクレイピング実行日を設定し、コード中にその旨のコメントを残す(捏造しない、docs/design/00-constitution.mdの精神を準用)。

### 4. 全文検索はSQLite FTS5のtrigramトークナイザを使用
日本語は単語間にスペースがなく、既定のunicode61トークナイザでは分割できない。3文字ごとに区切るtrigramトークナイザ(SQLite 3.34+に標準搭載)を採用し、部分一致検索を実現した。検索対象は現時点で議案タイトル・議案番号・会期名のみ(PDF本文抽出は未対応、将来課題)。

### 5. 検索結果のハイライトはHTMLタグではなく制御文字で表現する
FTS5の`snippet()`関数は任意の開始・終了マーカーを指定できるが、`<mark>`のようなHTMLタグを直接使うと、原本タイトルに万一HTML的な文字列が含まれていた場合にフロントエンドでの安全な描画(XSS対策)が難しくなる。代わりに制御文字(U+0001 / U+0002)をマーカーに使い(`packages/shared-types/src/search.ts`の`SNIPPET_HIGHLIGHT_START`/`END`)、フロントエンドはこれをもとにReact要素を組み立てる(`apps/web/src/components/HighlightedSnippet.tsx`)。`dangerouslySetInnerHTML`は使用しない。

### 6. ページネーションは「本質的に無制限なもの」にのみ適用する
`bills`/`meetings`はカーソルベースページネーションを実装した([[0005-scale-assumption]]の方針通り)。一方`legislators`は現職60議席という実世界の上限が明確にあるため、ページネーションを実装しない(YAGNI)。両者を区別する判断基準として記録する。

### 7. 議案一覧・議案検索結果には原本PDFの直リンク(sourceUrl)を併記する
内部キャッシュ(storagePath)ではなく、さいたま市サイトの原本URLに直接リンクする。著作権上「無断転載」に当たらないよう、自サイトでPDFを再配布しない設計とする(docs/adr/0002の利用規約配慮を実装レベルで徹底)。

## 結果(Consequences)

**メリット**
- 全て実データ(実際にさいたま市議会サイトから取得した59名の議員・7会派・1会期・38議案)で動作確認済み
- XSS対策とAIコンテンツの「捏造禁止」原則を、検索結果の表示という新しい局面でも一貫して適用できた

**デメリット**
- PDF本文を対象とした全文検索はまだ実現していない(タイトルのみ)
- 議員の会派履歴は「いつ観測したか」の記録に留まり、正確な加入日の遡及はできない

**将来の拡張性**
- PDF本文抽出(Phase3のAIパイプラインと連携、テキスト抽出後にsearch_indexへ追加)
- 会派履歴の精度向上(過去の会派別ページのアーカイブ等が見つかれば遡及可能)
