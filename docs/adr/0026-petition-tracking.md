# ADR-0026: 請願のトラッキング

## ステータス
承認済み(2026-07-10、ユーザーヒアリングにて決定)

## コンテキスト
ユーザーから「他にできることないか」との相談を受け、「みんなの国会」(https://minna-no-kokkai.com/)を参考に、市民発の提案(請願)・議員発の政策提言(意見書)を追う機能を提案した。事前調査で以下を実データで確認した。

- 会議録(発言の逐語記録)はrobots.txtで禁止されており対象外(docs/adr/0025のやり取りを参照)。
- 一方、**請願**(市議会議員の紹介を要する市民からの要望)は、既存の議案スクレイピングで使っている資料検索システム(Discuss Cabinet、docs/adr/0016)の「本会議」キャビネット内に、`年 > 定例会 > 日付フォルダ`配下の**「請願文書表」**(請願ごとの全文: 件名・請願者・紹介議員・付託委員会・要旨・理由)、`年 > 定例会 > 審議結果`フォルダ配下の**「請願審議結果一覧」**(請願番号・受理日・審議結果・議決日)として既に存在することを確認した。令和7年度の4定例会全てで同じフォルダ構成であることも確認済み。
- 「請願文書表」は表形式ではなく、請願ごとに「付託委員会名 → 請願番号/受理年月日 → 件名 → 請願者 → 紹介議員 → 要旨」という**縦に並ぶラベル付きテキスト**であり、pdf-parseの既定のテキスト抽出(座標なし)でも列が入れ替わらず読める(予算表(docs/adr/0024)のような列スクランブル問題が起きない)。
- 「請願審議結果一覧」も同様に、請願ごとに「番号→受理日(2行)→件名→審議結果→議決日(2行)」が上から順に並ぶ一連のテキストであり、座標なし抽出で読める。
- 紹介議員は1件の請願に複数人つくことがある(実データで確認: 「久保 美樹　金子 昭代」)。

ユーザーとスコープを協議し、以下を決定した。

## 決定

### 1. v1スコープ: 請願のみ(意見書・陳情は対象外)
「みんなの国会」に一番近いのは意見書(国等への政策提言)だが、まずデータ構造がシンプルで実データ検証が済んでいる請願から着手する。意見書(議員提出議案)・陳情(紹介議員不要な簡易版請願)は将来の拡張候補とする。

### 2. 紹介議員は既存Legislatorと名寄せする(多対多)
請願1件に紹介議員が複数つきうるため、`Petition`と`Legislator`の中間テーブルで多対多にする。名寄せは表決態度PDFの解析(docs/adr/0017 `voteStanceParsing.ts`)と同じ方式(空白除去した完全一致)を用いる。一致しない場合は捏造せずリンクなしのままにする。

### 3. 新規スクレイパーは既存のDiscussCabinetSessionを再利用する
`saitamaDeliberationResultsScraper.ts`と同じ`DiscussCabinetSession`を使い、対象文書のタイトルパターンを変えるだけの新しいスクレイパー(`saitamaPetitionScraper.ts`)を追加する。「請願文書表」は同一定例会内で複数回(追加分「その２」等)に分かれて配布されることがあるため、日付フォルダを全て走査してタイトルに一致する文書を全て収集する。

### 4. パーサーは座標なしテキスト抽出で十分(委員会審査結果報告一覧の`parseDeliberationResultText`とは別実装)
既存の`domain/bill/deliberationResult.ts`は「委員会審査結果報告一覧」(議案/請願が混在する別の文書)向けであり、そのまま請願専用の詳細情報(要旨等)を取るのには使えない。「請願文書表」「請願審議結果一覧」それぞれに専用のパーサーを`domain/petition/`に新設する。文書表は「請願番号 N 受理年月日」という区切りパターンでブロック分割し、各ブロック内をラベル(件名/請願者/紹介議員/要旨)で切り出す。

### 5. 審議結果の状態は実データにある文言をそのまま列挙する
実データで確認した「不採択」「取下げ」に加え、一般的に存在しうる「採択」「継続審査」を状態として持つ。未知の文言は捏造を避けるため`unconfirmed`扱いにする(docs/adr/0016と同じ方針)。「請願文書表」はあるが「請願審議結果一覧」にまだ載っていない請願(審議中)は`pending`とする。

## 結果(Consequences)

**メリット**
- 新規スクレイパーは既存のDiscussCabinetSession基盤を再利用でき、実装範囲がパーサー・取り込みロジック・表示に閉じる。
- 請願の全文(要旨・理由)まで取得できるため、単なる一覧以上の価値(市民が何を求めているかがそのまま読める)を提供できる。
- 紹介議員の名寄せにより、既存の議員詳細ページ(docs/adr/0020)と自然に連携できる。

**デメリット・今後の課題**
- 意見書(議員提出議案)・陳情は対象外。将来別ADRで検討する。
- Discuss Cabinetの本会議キャビネットの最古のフォルダは平成29年(2017年)だが、v1のバックフィル範囲は実装時の動作確認で扱った範囲に限る(全件バックフィルは別途の判断とする)。
