# ADR-0022: クロス集計のリッチ化(会派別ロールアップ・ドリルダウン)

## ステータス
承認済み(2026-07-07)

## コンテキスト
ユーザーから「クロス集計をもっとリッチな機能にしてほしい」との要望があった(docs/adr/0020の要望(2))。要望自体は抽象的だったため、ADR-0019で「今後の課題」として残していた2点を実装することにした:
- 議員数×タグ数が増えると表が横に長くなる問題への対応(会派単位のロールアップ表示)
- セル・見出しから元データへのドリルダウン

## 決定

### 1. 会派別ロールアップを既存の議員別マトリクスとは別の集計・別エンドポイントにする
`buildLegislatorTagMatrix`(議員単位)とは別に、`buildFactionTagMatrix`(会派単位)を`domain/vote/factionTagMatrix.ts`に新規実装した。既存の議員別集計ロジックを条件分岐で複雑にするのではなく、同じ入力(`VoteWithBillInfo[]`・タグマップ・status/meetingIdフィルタ)を受け取る独立した純粋関数として実装し、それぞれ単体でテストできるようにした。APIも`GET /cross-tab/faction-tags`を別途追加した(既存の`GET /cross-tab/legislator-tags`と対称)。

会派名が不明(無所属)の議員はまとめて「無所属」行にする(議員別マトリクスの表示方針と統一)。

### 2. `/analysis`ページに「議員別/会派別」の表示切替を追加する
クエリパラメータ`view`(`legislator`|`faction`、省略時は`legislator`)で切り替える。既存のstatus/meetingIdフィルタと直交する軸として扱う(3つのフィルタ状態をすべてURLで表現できるようにし、リンクの共有・ブックマークが可能)。

### 3. タグの列見出しから、そのタグ(・選択中の会期)に絞り込んだ議案一覧へドリルダウンできるようにする
タグ列見出しを`/search?tag=X&meetingId=Y`へのリンクにした。検索ページには既にタグ・会期フィルタがある(docs/adr/0018, 0021)ため、新しいエンドポイントを作らずに既存の検索ページへ寄せることができた。

### 4. 議員別ビューの議員名から、議員詳細(活動記録)ページへドリルダウンできるようにする
docs/adr/0020で追加した`/legislators/[id]`には、その議員の全投票記録(タグ・議案タイトル付き)が既にあるため、セル単体の詳細ページを新設するのではなく既存ページへリンクした。

## 結果(Consequences)

**メリット**
- 実データに安全なテスト用タグを一時的に付与して検証: 会派別ロールアップで、同一会派に属する複数議員の賛成/反対件数が正しく合算されること(自民党市議団の17件など)を確認した。議員別ビューの議員名リンク・タグ見出しの検索リンクも正しいURLが生成されることを確認した(検証後にテストデータは削除済み)。
- 回帰テストを追加(`factionTagMatrix.test.ts`、4件): 会派単位の合算・無所属へのまとめ・タグ未確定議案の除外・status/meetingIdフィルタを検証。

**デメリット・今後の課題**
- セルそのもの(特定議員×特定タグ)から「その議員のそのタグに関する投票のみ」に絞り込んだ表示は用意していない(議員詳細ページは全投票記録を表示する)。必要になれば議員詳細ページ側にタグでの絞り込みを追加する形で対応できる。
- 会派別ビューでは会派に投票記録データが無い場合(例: 会派結成前)の扱いは特に考慮していない(現状データでは問題にならないため)。
