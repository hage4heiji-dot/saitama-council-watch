# ADR-0016: 議案審議結果の同期(資料検索システムからのBill.status反映)

## ステータス
承認済み(2026-07-07)

## コンテキスト
Phase1のBillスクレイピングは「市長提出議案」ページから議案を取得するのみで、実際に本会議・委員会でどう審議されたか(可決/否決/継続審査等)は追っていなかった。このため公開サイト上ではすべての議案が`status=SUBMITTED`(提出)のまま表示され続けており、既に可決済みの議案でも「まだ審議されていないように見える」という利用者にとって誤解を招く状態だった。

審議結果の原本として2つの候補を調査した。

1. **さいたま市議会会議録検索システム(kaigiroku.net)**: 本会議の発言録。自然文からの審議結果抽出は解析コストが高く、誤読のリスクも大きい。
2. **さいたま市議会資料検索システム(Discuss Cabinet, `discusscabinet.net/saitama`)**: 「委員会審査結果報告一覧」「議案審議結果一覧」という、委員会ごとに議案番号と審査結果(原案可決/否決/継続審査等)が構造化された一覧PDFを公開している。

後者の方が構造化されており誤読のリスクが低いため採用した。実サイトを手作業で確認した結果、ログインフォームは`userid`/`password`が空値の`hidden` inputであり、実際の認証情報なしでゲストアクセスできることを確認した。

### 技術的な発見・詰まった点

- Discuss Cabinetはリンクではなく、フォームPOST(`/saitama/list`等)+ Cookieセッションで遷移するJS駆動のUI。各レスポンスに含まれる`hidden`フィールドを**全て**抽出し、遷移に必要な項目(`folder_id`, `move`, `docid`, `fileid`等)だけを上書きして次のリクエストに引き継ぐ方法でのみ正しく遷移できる。一部のフィールドだけを手動で組み立てると「エラー画面」が返ることを確認済み。
- フォルダ遷移の`move`パラメータ("down"/"up"/"")はサーバー側でセッション追跡されるパンくずスタックに対する相対操作であり、`folder_id`の値自体はほぼ確認用。兄弟フォルダへ移るには一度`up`で親へ戻ってから`down`で入り直す必要がある。
- `robotsCheck.ts`に**User-agentグループを無視するバグ**があった。`discusscabinet.net`の実際のrobots.txtは`User-agent: GPTBot` グループにのみ`Disallow: /`を持つが、既存実装はUser-agentのグループ分けを無視して全ての`Disallow`行を集めていたため、無関係なボット向け禁止規則で自分たちのアクセスまで誤って禁止と判定していた。`User-agent: *`グループのみを見るよう修正した([[../../apps/api/src/infrastructure/scraper/robotsCheck.ts]])。
- 委員会審査結果報告PDFはpdf-parseで抽出すると、テーブルが行単位ではなく**列(議案番号列→件名列→審査結果列)ごとにまとめて**抽出される。単純な行N↔行Nの対応付けはできないため、「項番号の並び」と「結果キーワードの並び」をそれぞれ収集し、件数が一致する区間だけを対応付ける方式にした。件数が一致しない区間は捏造を避けるため解決しない([[../../apps/api/src/domain/bill/deliberationResult.ts]])。
- 議案番号の表記が文書種別によって異なる: 委員会審査結果報告は「議案第１１２号」のように種別接頭辞が必須。専決処分の「議案審議結果一覧」は接頭辞なしの「第106号」のみ。当初、接頭辞を任意にした単一の正規表現を使ったところ、件名中の「（第１号）」(補正予算の回次表記であり議案番号ではない)を誤って議案番号と認識し、予算委員会の4議案が解決不能になるバグを実データで発見した。文書種別ごとに正規表現を分離して解決した。

### 対象範囲(スコープ)
令和8年6月定例会の全38議案について実データを手作業で確認した結果、29議案(第106〜132号、第138〜139号)は委員会審査結果報告・議案審議結果一覧のいずれかに審査結果が記載されていたが、残り9議案(第133〜137号: 人権擁護委員候補者の推薦、第137号: 固定資産評価員の選任、第140〜141号: 教育委員会教育長・委員の任命、第142〜143号: 監査委員の選任)は人事同意案件であり、委員会付託を経ないため審査結果の記載が見当たらなかった。

## 決定

- `BillStatus`に`UNCONFIRMED`(詳細要確認)を追加する。会期が終了しているにもかかわらず原本から審議結果を特定できなかった議案は、捏造を避けるため「提出」のまま放置せず明示的にこのステータスにする(ユーザー指示: 「ステータスが確認できないものは詳細要確認などステータスを正しくして」)。
- 既に確定済み(`passed`/`rejected`/`carried_over`)のステータスは、たまたま今回の原本に見当たらなかったという理由だけで`unconfirmed`へ後退させない(確認済みの事実を消さない)。
- 審議結果の同期対象は「会期(本会議)が終了している(`endDate < 実行時刻`)」もの全件とし、`令和8年6月定例会`に限定するハードコードはしない(将来の会期にもそのまま使える)。ただし全議案が確定済み/unconfirmedになった会期は再スクレイピングをスキップする(ADR-0002の負荷配慮、および無駄なアクセスを避けるため)。
- 審査結果の原文(原案可決/可決/承認 → passed、原案否決/否決 → rejected、継続審査/継続審議 → carried_over)以外の文言(例: 取下げ)は、現行`BillStatus`で正しく表現できないため`unconfirmed`にフォールバックする(断定できないものを断定しない)。
- 委員会審査結果報告・議案審議結果一覧のPDF自体は`Document`テーブルには保存しない(スコープ限定の判断。下記「今後の課題」参照)。バッチジョブの標準出力に処理件数を出力するのみ。

## 結果(Consequences)

**メリット**
- 実データで検証した結果、38議案中29議案が正しく`passed`に、9議案が`unconfirmed`(詳細要確認)になることを確認した。ステータスが誤って「提出」のまま放置される議案がなくなった。
- `robotsCheck.ts`のUser-agentグループ無視バグを修正したことで、他サイトへの将来の適用でも誤検知しなくなった。
- 会期終了・全議案確定済みの場合は自動的に再スクレイピングをスキップするため、cronで恒久的に走らせても資料検索システムへの負荷は最小限。

**デメリット・今後の課題**
- 委員会審査結果報告・議案審議結果一覧PDF自体は`Document`として保存していないため、「どの原本からこのステータスになったか」をDB上で追跡できない(ログ出力のみ)。将来的に`Bill`へ`statusSourceDocumentId`のような列を追加し、原本を保存・リンクすることを検討する。
- 議案の審査結果表記(委員会審査結果報告 vs 議案審議結果一覧)は現行2種類のみ実データで確認済み。将来別の書式(例: 臨時会特有の書式)が出てきた場合はパーサーの拡張が必要。
- 請願(`請願第N号`)の審査結果も同じPDFから取得できるが、本アプリは請願をモデル化していないため現時点では捨てている。将来請願を追跡する場合は`parseDeliberationResultText`が返す`kind: "請願"`の項目を活用できる。
