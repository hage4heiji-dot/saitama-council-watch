# ADR-0020: 議員詳細(活動記録)ページの追加

## ステータス
承認済み(2026-07-07)

## コンテキスト
ユーザーから以下の要望があった(要約):
1. 検索・クロス集計に期間の絞り込みがない
2. クロス集計をもっとリッチにしてほしい
3. 市議会の年間のマイルストーンがほしい
4. 過去の議員を含めて任期がわかるようにしてほしい
5. **最も重要**: その議員がどういった活動をしたのかがわかるようにしたい。そうしないと選挙で続投判断ができない。

5点のうち(5)が明示的に最優先とされたため、本ADRではまず(5)を実現する「議員詳細(活動記録)ページ」を追加する。(1)〜(4)は別途対応する(後述)。

(4)の「過去の議員を含めた任期」について事前に調査したところ、さいたま市議会公式サイトには以下の情報のみが存在し、個々の議員の在任期間(いつからいつまで在職したか)を示すデータは存在しなかった:
- 現職名簿ページ(例: `city.saitama.lg.jp/gikai/001/meibo/`)は現職のみを掲載し、当選回数は載っているが個々の任期日付はない
- 現在の任期満了日(2027-04-30、令和5年4月9日の統一地方選挙に基づく4年任期)は市サイトに公開されている単一の事実として存在する
- 過去の選挙結果ページは選挙単位で構成されており、個人のキャリア(いつからいつまで在職)を再構成するには複数回の名簿差分が必要で、現時点ではスコープ外とする

捏造しないという原則(docs/design/00-constitution.md)に基づき、個々の議員の historical な在任期間は今回実装しない。当選回数の取得・任期満了日の表示は別フェーズの課題として残す。

## 決定

### 1. 既存データ(投票記録・議案・タグ)のみで「活動記録」を構成する
新しいスクレイピングは不要。`Vote`(議員×議案の投票記録、docs/adr/0017)と、承認済みAIタグ(docs/adr/0007)を組み合わせ、議員ごとに「いつ・どの議案に・どう投票したか」の一覧を表示する。これにより「続投判断の材料」を、実在するデータの範囲で提供する。

### 2. `LegislatorRepository`を拡張し、元議員も取得可能にする
`findAll()`は`isActive: true`をハードコードしたフィルタを持っていた(元議員が一律非表示になっていたバグ)。`findAll(options?: { includeInactive?: boolean })`に変更し、`findById`・`findFactionHistory`を追加した。API側は`GET /legislators?includeInactive=true`で元議員も含める。

### 3. `VoteRepository.findByLegislatorId`を追加する
既存の`findByBillId`(議案→投票者一覧)の逆方向、議員→投票した議案一覧を返す。議案のタイトル・号数・状態・sourceDocumentIdを併せて返し、アプリケーション層(`buildLegislatorDetail`)でsourceUrlとタグを付与する(`buildBillDetail`と同じパターン)。

### 4. 議員詳細DTO(`LegislatorDetail`)は「プロフィール + 会派異動履歴 + 投票サマリー + 投票記録一覧」を持つ
`GET /legislators/:id`が返す。投票記録には議案のタグを含めるため、投票した議案から見えるトピック傾向も把握できる。

## 結果(Consequences)

**メリット**
- 実データ(令和8年2月・3月定例会の投票記録)で動作確認済み。ある議員(36件全て賛成)の詳細ページで、会派異動履歴・投票サマリー・投票記録一覧(議案タイトル・状態・投票日・表決態度・タグ)が正しく表示されることを確認した。
- 元議員も`?includeInactive=true`で一覧に表示できるようになった(ただし現時点で実際に元議員のデータが投入されているわけではなく、`isActive=false`のレコードが登録されればそのまま機能する)。

**デメリット・今後の課題**
- 当選回数・個々の任期日付は未実装(実データが存在しないため)。当選回数はスクレイピング可能なため、別PRで追加を検討する。
- 検索・クロス集計の期間フィルタ、クロス集計の拡張、年間マイルストーンページは本ADRの対象外(ユーザー要望の(1)〜(3)、別途対応)。
