# ADR-0024: 歳出予算の款別内訳可視化(「お金の流れ」v1)

## ステータス
承認済み(2026-07-08、ユーザーヒアリングにて決定)

## コンテキスト
ユーザーから「コンテンツを充実させていきたい」との要望を受け、既存のDBスキーマに存在するが未実装だった`Budget`モデルを軸に「お金の流れ」機能を提案した。事前調査で以下を実データで確認した。

- さいたま市議会サイトの「予算の概要」ページ(`/006/007/011/001/...`)は金額をHTML本文に持たず、全て添付PDF(数百ページの予算書)に依存しており、直接スクレイピングする対象として扱いにくい。
- 一方、**予算関連の議案(当初予算・補正予算)は既に`saitamaBillsScraper`で議案PDFとして原本保存済み**であり(`Bill.sourceDocumentId` → `Document.storagePath`)、新規スクレイパーは不要と判明した。
- 一般会計・特別会計の議案PDFは「第１表 歳入歳出予算(補正)」という款(かん)/項単位の金額表を持ち、構造が安定している(実データ: 議案第18号 令和8年度一般会計予算、議案第108号 令和8年度一般会計補正予算(第1号)等で確認)。
- 一方、公営企業会計(水道事業会計・下水道事業会計・病院事業会計)は「収益的収入及び支出」「資本的収入及び支出」という全く別の複式簿記形式であり、同じパーサーを使い回せない(実データで確認)。

ユーザーとスコープを協議し、以下を決定した。

## 決定

### 1. v1スコープ: 歳出(支出)の款別内訳、普通会計(一般会計+特別会計)のみ、当初予算+補正予算を反映
- 歳入(収入)は対象外(将来拡張)。
- 公営企業会計(水道・下水道・病院事業会計)は表形式が全く異なるため対象外とし、別ADR・別PRで扱う。
- 当初予算を基準に、同一会計年度・同一会計(一般会計/各特別会計)内の補正予算を提出日順に適用し、款ごとの**最新の金額**を算出する(「積み上げ」)。補正予算の表は変更のあった款のみを含む(実データで確認: 全款が毎回載るわけではない)ため、単純に最新1件の議案だけを見ればよいわけではなく、当初予算をベースラインとして款ごとに上書きしていくマージ処理が必要。

### 2. 新規スクレイパーは作らず、既存Bill/Documentを再利用する
予算議案は`saitamaBillsScraper`により既に`Bill`・`Document`(PDF原本)として取り込まれている。`Bill.title`が予算関連のパターンにマッチするものを抽出し、そのPDFを新しいドメインパーサーで解析してバックフィルする方式にする。

### 3. 会計種別の判定はタイトル文字列パターンで行う(実データで全パターンを確認済み)
- `〇〇特別会計` を含むタイトル → 普通会計(特別会計)として対象に含める(例:「食肉中央卸売市場及びと畜場事業特別会計」のように「事業」を含むが「特別会計」表記であるものも普通会計側)。
- `〇〇事業会計`(「特別」を含まない)→ 公営企業会計として**除外**(水道事業会計・下水道事業会計・病院事業会計の3種のみ実データに存在)。
- タイトルに「予算」を含まない議案(専決処分の一般的な報告等、`Bill`全体)は対象外。「専決処分の報告及び承認を求めることについて(一般会計補正予算)」のような専決処分による補正は対象に含める(実質的に一般会計補正予算のため)。

### 4. パーサーはX/Y座標付きテキスト抽出(`extractPositionedPdfText`)を使う
`pdf-parse`の既定のテキスト結合では表の列位置情報が失われる(表決態度PDFの解析、docs/adr/0017と同じ問題)。今回も款/項の行番号列・金額列がX座標クラスタとして分離できることを実データで確認したため、座標情報を使って列を判定する。

補正予算の金額列(補正前の額/補正額/計)の境界検出は、当初はデータ行の金額のX座標から動的に推定する方式(ギャップ検出・出現頻度)を試したが、款が1つしかない小規模な議案(例: 国民健康保険事業特別会計補正予算)ではデータ行数が少なすぎて誤検出することが実データで判明した。最終的に、**ヘッダー行自体(「款/項/補正前の額/補正額/計」)の項目位置**から列境界を求める方式に変更した。ヘッダー行はデータ行数によらず必ず1回だけ存在するため、款が1つしかない議案でも安定する。

また、歳出表の開始を示す見出し文字列にも表記揺れがあることが実データで判明した(「　歳　出」(先頭にも全角スペース)と「歳　出」(先頭なし)の両方が存在)。空白除去後に「歳出」と一致し、かつ地の文中の詰まった「歳出」(空白なし)ではないものを見出しとみなすことで両方に対応した。

### 5. `Budget`スキーマに`accountName`(会計名)を追加する
既存スキーマは`(fiscalYear, category)`に相当する一意キーを想定していなかったが、特別会計ごとに款の名称(「総務費」等)が重複しうるため、会計(一般会計/各特別会計)を区別する`accountName`列が必須と判断した。`@@unique([fiscalYear, accountName, category])`でupsertする。

### 6. `description`には項単位の内訳をそのまま格納する(捏造しない)
`Budget.description`(必須)には、款の下位にある項(サブカテゴリ)の名称と金額をPDFから抽出したまま連結して格納する(例:「総務管理費 36,419,261千円/企画費 15,832,470千円/...」)。ユーザー向けの説明文をAIやコードで新たに生成することはしない(docs/design/00-constitution.mdの「捏造禁止」原則)。

### 7. 表示は款別の横棒グラフ(単一系列、直接ラベル)とする
「歳出がどこに使われているか」という単一指標(金額)のカテゴリ別比較のため、dataviz skillの手順に従い横棒グラフを採用。単一系列なので凡例は不要、`--sequential-450`(青)を使う。年度セレクタで会計年度を切り替えられるようにする(複数年度のデータが貯まった際の比較に備える)。

## 結果(Consequences)

**メリット**
- 新規スクレイパーが不要なため、実装範囲がパーサー・取り込みロジック・表示に閉じる。
- 既に保存済みのPDF原本を使うため、原本参照(`relatedBillId`)による裏付けが自然に確保される。

**デメリット・今後の課題**
- 歳入・公営企業会計は今回のスコープ外(将来別ADRで対応)。
- 補正予算の「積み上げ」ロジックは款単位の上書きであり、項単位の変化までは追跡しない(v1では款単位の可視化のみが目的のため十分と判断)。
- 過去年度分は既存DBに保存済みの議案PDFの範囲でのみ遡れる(令和7年度以降の一部のみ)。
