# ADR-0006: 公開用AI生成コンテンツはPostgreSQL、技術的キャッシュのみSQLite

## ステータス
承認済み(2026-07-06、アーキテクト提案をユーザーが採用「任せる」)

## コンテキスト
プロジェクト憲法([[../design/00-constitution.md]])では「SQLite = AI専用(キャッシュ/Embedding/全文検索/一時データ/AI解析結果)、業務データを持たない」と定義されていた。この定義通りに「AI解析結果(要約・タグ・FAQ)」までSQLiteに置くべきか、アーキテクト側から再検討を提案し、ユーザーの判断を仰いだ。

## 検討した選択肢

### 案A: 憲法の記載通り、AI解析結果も含めてすべてSQLite
- メリット: SQLiteをAI専用として完全隔離でき、AIモデル変更の影響がPostgresに一切及ばない
- デメリット:
  - 市民向けページで「議案の事実(Postgres)」と「AI要約(SQLite)」を同時表示する際、DBをまたいだアプリ層JOINが常に必要
  - SQLiteファイルはPostgresのようなレプリケーション/PITRの仕組みが薄く、誤削除・ディスク障害でサイトの主要コンテンツ(AI要約)が丸ごと消えるリスクがある
  - バックアップ・監査ログが二重管理になりやすい

### 案B(採用): 性質による二分
AI由来データを「市民に公開する最終成果物」と「純粋に再生成可能な技術層」に分け、保存先を変える。

| 性質 | 保存先 | 理由 |
|---|---|---|
| 公開用AI生成コンテンツ(要約・タグ・FAQ) | **PostgreSQL**(`ai_contents`テーブル) | 事実データとJOINして表示する製品コンテンツ。来歴を外部キー制約で強制することが「AIは事実を創作しない」の技術的裏付けになる。バックアップ・監査対象として業務データと同格に扱うべき |
| 埋め込みベクトル、全文検索インデックス、LLM応答キャッシュ、スクレイピング/パースの中間状態 | **SQLite** | 失っても「Postgresの原本参照 + 再実行」で完全復元できる、文字通りのキャッシュ |

## 決定
案Bを採用する。`ai_contents` テーブル(PostgreSQL)は以下を必須カラムとする:

- `source_document_id` (NOT NULL, FK → `documents.id`) — 原本なきAI生成をDB制約レベルで禁止
- `model_version`, `prompt_version` — モデル/プロンプト変更時の追跡・再生成・監査を可能にする
- `generated_at`
- `is_verified` (boolean) — 人手確認ゲート([[0007-ai-human-review-gate]])との連携

SQLiteには業務データを一切置かない、という憲法の原則はこの案の方がむしろ厳密に守られる(SQLiteの中身は完全に「再生成可能なキャッシュ」のみになるため)。

## 結果(Consequences)

**メリット**
- Postgres単体で「事実 + 公開AIコンテンツ」の一貫したJOINクエリが可能
- バックアップ戦略が一本化され、SPOFである単一VPSでのデータ消失リスクを下げる([[0003-hosting-single-vps-docker]])
- `model_version`/`prompt_version` により、AIモデル変更時の影響範囲を「再生成すればよい」という形で最小化できる(保守方針と整合)

**デメリット**
- SQLiteとPostgresの両方に「AI関連データ」が分散するため、初見では境界がわかりにくい → 本ADRと憲法への参照コメントをコードに残すこと

**将来の拡張性**
- 埋め込み検索がSQLite(`sqlite-vec`)で性能不足になった場合、PostgreSQL + `pgvector` へ段階的移行が可能(`ai_contents`側の設計には影響しない)
