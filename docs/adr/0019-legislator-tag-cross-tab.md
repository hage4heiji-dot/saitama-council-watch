# ADR-0019: 議員×タグ クロス集計ページの追加

## ステータス
承認済み(2026-07-07)

## コンテキスト
ユーザーから「議員と議案と可決かどうかと賛成かどうかとタグでクロス集計できるページ」の追加要望があった。候補として(1)議員×タグのマトリクス、(2)会派×タグのマトリクス、(3)投票1件=1行の一覧、の3案を提示したところ、(1)議員×タグのマトリクス(セルに賛成/反対件数、可決/否決で絞り込み可能)を選んだ。

## 決定

### 1. 集計はアプリケーション層(ドメイン純粋関数)で行う
既存の`tagCounts.ts`/`billTags.ts`と同じ方針で、Postgresから「全投票記録(議員名・会派名・議案の状態・sourceDocumentId付き)」と「承認済みタグ」をそれぞれ取得し、`domain/vote/legislatorTagMatrix.ts`の純粋関数`buildLegislatorTagMatrix`で議員×タグの賛成/反対件数に集計する。SQLでの複雑な集計クエリは書かず、既存データ取得パターン(全件取得→アプリ側で集計)を踏襲した。

### 2. タグが確定していない議案・投票記録のない議員は結果から除外する
承認済み(is_verified=true)のタグが1つも無い議案は集計対象から除外する(捏造しない、docs/adr/0007)。また、投票記録が1件もない議員は行として出力しない(59名全員を機械的に列挙して大半が空欄になるより、実データがある行だけを見せる方が「まだデータが無い」ことが分かりやすいと判断した)。

### 3. `可決かどうか`はクエリパラメータ(`status`)によるフィルタとして実装する
「可決かどうか」と「賛成かどうか」は独立した軸(可決した議案でも反対した議員はいる)であるため、`status`(議案のBillStatus)は絞り込み条件、賛成/反対はセルの値、タグは列、議員は行、という役割分担にした。

### 4. セルは賛成/反対の件数のみ表示する(欠席・棄権は表示しない)
UIのシンプルさを優先し、欠席・棄権はいったん表示対象外とした。将来必要になれば`LegislatorTagCell`に`absent`/`abstain`を追加する形で拡張できる。

## 結果(Consequences)

**メリット**
- 実データ(令和8年2月定例会の投票記録2,052件)に安全なテスト用タグを一時的に付与して検証し、57名の議員(投票記録のある全員)について、議員×タグの賛成/反対件数が正しく集計されることを確認した。`status`フィルタ(すべて/提出/可決等)についても、フィルタ条件に応じて正しく行数が変わることを確認済み。
- 回帰テストを追加(`legislatorTagMatrix.test.ts`): 集計・タグ未確定議案の除外・statusフィルタ・投票記録のない議員の除外、の4点を検証している。

**デメリット・今後の課題**
- 本番では現時点で承認済みのAIタグが0件のため、このページはまだ空の状態(「現在、承認済みのAIタグと投票記録の両方が判明している議案がありません」)になる。管理画面でのタグ承認が進み、かつ投票記録のある議案とタグが確定した議案が重なるまでは実データが表示されない。
- 議員数(59名)×タグ数が増えると横に長い表になる。現状は`overflow-x-auto`で対応しているが、タグ数が大きく増えた場合は列の絞り込みUIが必要になる可能性がある。
