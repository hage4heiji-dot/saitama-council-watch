# ADR-0007: AI生成コンテンツの公開前に人手確認ゲートを設ける

## ステータス
承認済み(2026-07-06、アーキテクト提案をユーザーが「許容できる」として採用)

## コンテキスト
プロジェクト憲法の絶対原則「AIは事実を創作してはいけない。元データが存在しない内容は生成禁止」([[../design/00-constitution.md]])をプロンプト指示だけで保証するのは不十分であり、政治・行政という性質上、誤情報がそのまま公開されるリスクは特に重い。

## 検討した選択肢
1. 機械的なグラウンディングチェックのみ(固有名詞・数値の原本突合)
2. 機械的チェック + Claudeによる自己検証パス(要約が原文で裏付けられるかを再度問う)
3. 上記に加え、新規/更新分のみ人手確認してから公開(採用)

## 決定
`ai_contents.is_verified = false` の状態では公開Webに表示しない。管理画面で人手確認(新規追加分のみ、全件ではなく差分確認)を経て `is_verified = true` になったコンテンツのみ公開する。

自己検証パス(別セッションのClaude呼び出しで「原文だけで裏付けられるか」を問い直す)でフラグが立った場合は、優先的に人手確認に回す。

## 結果(Consequences)

**メリット**
- 「AIは事実を創作しない」という憲法上の絶対原則に対する、最後の技術的・運用的な砦になる
- 誤情報公開時の説明責任(誰がいつ確認し公開したか)を監査ログとして残せる([[../design/01-basic-design.md]] §8)

**デメリット**
- 運用コストが増える(人手確認の工数)。ただし差分確認のみのため、データ量に対してスケールする
- 確認が滞留すると公開が遅延するリスクがある → バッチのジョブ監視([[../design/01-basic-design.md]] §7)で未確認滞留を検知できるようにする

**将来の拡張性**
- 確認精度・実績が蓄積された段階で、特定カテゴリ(定型的な内容等)のみ自動公開に緩和する運用変更が可能(スキーマ変更は不要、運用ルールの変更のみ)
