# ADR-0013: Phase3(AIパイプライン)の設計判断

## ステータス
承認済み(2026-07-06)

## コンテキスト
Phase3で要約・分類・タグ付け・FAQ生成のAIパイプラインと、グラウンディング検証、管理確認画面を実装した。実装にあたり確定した判断をまとめる。

## 決定事項

### 1. PDFテキスト抽出には`pdf-parse`を使用
議案原本はPDFとして保存されている(docs/adr/0002)。Claudeへの入力は原本テキストのみに限定する(docs/design/00-constitution.md)ため、PDFからのテキスト抽出が必須の前段処理となる。ネイティブ依存のない`pdf-parse`を採用し、Phase1で実際に取得済みの議案PDF(38件)に対して抽出できることを検証済み(227〜15,661文字、全件非空)。画像PDF等でテキストが空になる場合は、捏造を避けるため生成をスキップする。

### 2. Claude APIの構造化出力はtool_choiceで強制する
自由形式のテキスト応答をパースするのではなく、Anthropic APIの`tools`+`tool_choice: {type: "tool", name: ...}`を用いて、要約・タグ・FAQを常に決まったJSON形状で受け取る。パース失敗によるリトライ処理の複雑化を避け、`docs/design/01-basic-design.md §6.1`の「構造化出力を強制」方針を実装レベルで満たす。

### 3. グラウンディング検証は二段構え(既存設計通り)
1. **機械的チェック**(`domain/ai/mechanicalGroundingCheck.ts`): 要約中の数値(金額・日付・条数等)が原文に存在するかを突合。日本語の固有名詞抽出には形態素解析が必要でありYAGNIの観点からPhase3のスコープ外とし、数値のみを対象とする(議案の性質上、数値の捏造がもっとも実害が大きいため優先)。
2. **自己検証パス**(`ClaudeBillAnalysisAdapter.checkGrounding`): 別のClaude呼び出しで「要約中に原文の裏付けがない主張があれば列挙せよ」と問い直す。

いずれかで問題が見つかった場合、`ai_contents.groundingNote`に理由を記録し、管理確認画面での優先レビュー対象として表示する(`is_verified`は問題の有無にかかわらず常にfalseで作成される。グラウンディング結果は「レビューの優先度」を示すものであり、それ自体が公開可否を決めるものではない)。

### 4. `ai_contents.grounding_note`カラムを追加
グラウンディング検証で問題が見つかった場合のみ設定するnullableカラム。管理者がレビュー時に「なぜ確認が必要か」を機械的チェック結果込みで把握できるようにする。

### 5. 管理確認画面はADMIN_API_TOKENによる仮保護
Phase4でOAuthベースの管理者認証を導入するまでの間、環境変数`ADMIN_API_TOKEN`と一致するリクエストのみを許可する(`requireAdminToken`ミドルウェア)。`/internal/ai-contents/*`は`/api/v1`とは別の名前空間とし、公開CORS対象・公開ナビゲーションから隔離する。フロントエンドの承認フォームは素のHTML `<form>` POSTで実装し、Next.jsのRoute Handler(`/admin/review/verify`)がサーバー側でトークンヘッダを付けてinternal APIを呼ぶことで、ブラウザから直接トークンヘッダを送信せずに済む。

### 6. LLM応答キャッシュ(SQLite)でコストを抑制
`llm_response_cache`テーブル(Phase0で設計済み)に、原本テキスト+プロンプト版のハッシュをキーとして応答をキャッシュする(`docs/design/01-basic-design.md §6.4`)。同一内容への再生成ジョブ実行時にAPIを再呼び出ししない。

### 7. 実際のClaude API呼び出しは今回のセッションでは実行検証していない
ユーザーの判断により、今回はAPI利用料金が発生する実行検証を行わなかった。`CLAUDE_API_KEY`はダミー値のままである。以下は実データで検証済み:
- PDFテキスト抽出(実際の議案PDF)
- 機械的グラウンディングチェック(フィクスチャデータでの単体検証: 捏造した数値を正しく検出)
- AiContentRepository・管理確認画面のAPI/UI(手動で挿入したテストデータによる疎通確認、`is_verified=false`のコンテンツが公開APIから除外されることを確認)

実際のAPIキーが用意され次第、`npm run --workspace apps/api generate:ai-content`で実データに対する検証を行うこと。

## 結果(Consequences)

**メリット**
- グラウンディング検証・人手確認ゲートという「AIは事実を創作しない」ための多層防御が、コードとして具体化された
- 管理確認画面と公開APIの両方で、未承認コンテンツが決して公開されないことを実データで確認済み

**デメリット**
- 実際のClaude応答の品質(要約の読みやすさ、タグの粒度等)はまだ検証していない
- 自己検証パスは追加のAPI呼び出しコストが発生する(要約生成の実質2倍)

**将来の拡張性**
- 実キー入手後、プロンプトの微調整(`BILL_ANALYSIS_PROMPT_VERSION`を上げて再生成)が既存データに影響を与えず行える
- 管理確認画面はPhase4のOAuth認証実装時に`requireAdminToken`を`requireAdminUser`的なミドルウェアへ差し替えるだけで移行できる設計にしてある
