# ADR-0008: バッチ/AI処理を`worker`コンテナとしてAPIから分離

## ステータス
承認済み(2026-07-06、アーキテクト提案をユーザーが「賛同する」として採用)

## コンテキスト
[[0001-architecture-style]] で選択したモジュラーモノリスは、単一デプロイ単位のシンプルさを優先する一方、「AIパイプラインの負荷が急増した場合、Web/APIと同一ホストでリソース競合が起きうる」という課題を残していた。スクレイピング・パース・AI呼び出しは実行時間が読めない重い処理であり、これをWebリクエスト処理と同一イベントループで回すと、バッチ実行中に市民向けページの応答が遅延するリスクがある。

## 決定
`api`(HTTPリクエスト処理専任)と `worker`(スケジュールジョブ実行専任)を、**同一Dockerイメージ・別プロセス・別Composeサービス**として分離する。コードベース・ビルドは共通化し、起動コマンドのみを変える。

## 結果(Consequences)

**メリット**
- Web応答性能とバッチ処理が分離され、片方の負荷がもう片方に波及しない
- イメージ共通化によりビルド・依存管理の二重コストはほぼゼロ
- 将来的に `worker` だけを別ホスト/別リソースサイズにスケールすることが、Compose定義の変更のみで可能

**デメリット**
- Composeサービス数が増え、初期構成がわずかに複雑になる

**将来の拡張性**
- 負荷が増大した場合、`apps/api/src/batch` を独立した `apps/worker` パッケージとして完全分離し、専用ジョブキュー(BullMQ+Redis等)へ移行する土台になる([[0001-architecture-style]] の「戦略的モノリス解体」の第一歩)
