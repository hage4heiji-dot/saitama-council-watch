# ADR-0002: データ取得はさいたま市議会公式サイトのスクレイピング

## ステータス
承認済み(2026-07-06、ユーザーヒアリングにて決定)

## コンテキスト
議事録・議案・会議録などの原本データをどう取得するかは、データフロー・バッチ設計・法務(著作権・利用規約)対応に直結する、初期段階で確定すべき前提だった。

## 検討した選択肢
1. さいたま市議会公式サイトのスクレイピング(採用)
2. 運営者による手動アップロード
3. 公式オープンデータ/APIの利用
4. ハイブリッド(スクレイピング + 手動補完)

さいたま市議会サイトには網羅的な公式オープンデータ/APIが存在しないため、選択肢3は現実的でないと判断。選択肢2は更新の即時性・運用コストで劣る。

## 決定
公式サイトのスクレイピングを一次データ取得手段とする。

## 結果(Consequences)

**満たすべき制約**
- robots.txt の遵守
- リクエスト間隔を空ける(サイトへの負荷配慮、politeness delay)
- 取得したHTML/PDFは改変せず原本として保存し、`source_url` と取得日時を必ず記録する
- サイト構造の変更に対する耐性は、スクレイパー実装を `infrastructure/scraper` 配下に閉じ込めることで局所化する([[0001-architecture-style]])

**デメリット**
- サイト構造変更時にスクレイパーが壊れるリスクは避けられない → バッチ実行監視([[../design/01-basic-design.md]] §7)で早期検知する

**将来の拡張性**
- 将来さいたま市が公式オープンデータ/APIを提供した場合、スクレイパーアダプタを差し替えるだけで移行可能
