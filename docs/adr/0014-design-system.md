# ADR-0014: 公開サイトのデザインシステム導入

## ステータス
承認済み(2026-07-06)

## コンテキスト
公開Webサイトの見栄え・コンテンツ充実にあたり、「みんなの国会」(https://minna-no-kokkai.com/)を参考にした。同サイトの調査で得た知見:

- トップページで「会期の進捗(%・残日数)」「法案の状況別件数」など生きた数字を強調している
- 法案詳細ページは要約→ポイント→詳細解説→提出者内訳という構成
- カテゴリ別・会派別に色分けした視覚的な分類がある

これらの**構成パターン**は参考にするが、配色や具体的なブランド要素をそのまま模倣するのではなく、自分たちの検証済みデザインシステムを別途構築する。

## 決定事項

### 1. 色はdataviz skillの検証済み参照パレットをそのまま採用
自前で配色を決めてCVD(色覚多様性)安全性やコントラストを目視確認するのではなく、`dataviz` skillが提供する検証済みパレット(`references/palette.md`)を採用し、`scripts/validate_palette.js`でライト/ダーク両モードを検証した(カテゴリカル8色・ステータス4色ともにPASS、一部警告は直接ラベル併記で緩和)。

### 2. コンポーネントは「データの仕事」で形式を選ぶ
- 会期進捗 → 単一の比率 → **Meter**(dataviz skill: choosing-a-form.md)
- 議案ステータス別件数 → 複数の見出し数字 → **KPI row(StatTile群)**
- 会派別議席構成 → 部分と全体 → **横積み上げバー**、色は会派名の辞書順で固定割当て(議席数の順位ではなく実体に紐づける。将来議席数が変わっても同じ会派は同じ色になる)
- 議案ステータス → **StatusBadge**(色は識別色と別系統の状態色。色だけに依存せずラベルを必ず併記)

### 3. コンテンツの充実は「今あるデータ」の範囲に限定
「みんなの国会」にある「会派別の賛否内訳」「SNS反応」「学ぶ」コンテンツ等は、対応する実データ(投票結果のスクレイピング、一般解説記事)がまだないため今回は追加しない。捏造・仮データでの見た目だけの充実はしない(docs/design/00-constitution.mdの原則を UI コンテンツにも適用)。

## 結果(Consequences)

**メリット**
- 色のアクセシビリティを「目視」ではなく検証スクリプトで担保できた
- コンポーネント(StatTile/Meter/StatusBadge/FactionBar)は今後増えるページでも再利用できる
- 「データの仕事に応じて形式を選ぶ」という判断基準が明文化され、今後のUI追加でも迷わない

**デメリット**
- ホームページの議案集計はAPIのページネーション上限(100件)を超えると不正確になる(現状38件のため未対応、YAGNI。コメントに明記済み)

**将来の拡張性**
- 投票結果(Vote)データが投入されれば、法案ごとの会派別賛否も同じFactionBar/StatusBadgeの仕組みで表示できる
- 100件を超える規模になったら、集計専用のAPIエンドポイントを追加する
