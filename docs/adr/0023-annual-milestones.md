# ADR-0023: 年間マイルストーン(会議日程一覧のスクレイピングと月次可視化)

## ステータス
承認済み(2026-07-08)

## コンテキスト
ユーザーから「市議会の年間のマイルストーンがほしい」との要望があった(docs/adr/0020の要望(3))。事前調査で、個々の議員の任期のような情報は公開されていない一方、**「会議日程一覧」**というページ(`https://www.city.saitama.lg.jp/gikai/002/teireirinji/kaiginitteiitiran/p032830.html`)には、直近1年分の本会議・委員会(予算委員会・決算特別委員会・5常任委員会・各特別委員会等)の開催日程が週単位で掲載されていることを確認した。ユーザーに「委員会スケジュールも新規スクレイピングして含める」か「既存データ(定例会の会期のみ)で作るか」を確認したところ、前者(スクレイピングして含める)を選んだ。

このページのHTML構造は事前に実機取得して詳細に調査済み(週見出し`<h2>`+直後の`<table>`、日付セルに年が無い、時刻・「散会後」で始まる行が新しい会議、そうでない行は委員会名の折り返しとして連結する必要がある、等)。

## 決定

### 1. 個別の日程を`CommitteeMeeting`という新モデルで持つ
既存の`Meeting`は会期(定例会・臨時会)全体の期間を表すモデルであり、1日単位の開催記録とは粒度が異なる(実際、`Meeting`モデルの元のコメントに「個別の本会議・委員会単位のより粒度の細かい日程は将来の別モデル化を検討する」と将来課題として明記されていた)。`committeeName`は原本の表記をそのまま保持し(「予算委員会（企業会計関係審査）【中継】」等、捏造・正規化のしすぎを避ける)、月次集計・表示向けに括弧書きを除いた`committeeBaseName`を別途計算して持つ。

`meetingId`は、日付が既存の`Meeting.startDate`〜`endDate`の範囲に収まる場合のみ設定し、判定できない場合はnullのままにする(捏造しない)。

### 2. パース処理は複雑な分岐を伴うため、pure functionとして切り出しテストを書く
「時間・会議名」セルは`<br>`区切りだが、時刻(「10時」「9時30分」)または「散会後」で始まる行のみが新しい会議の開始で、それ以外は委員会名が長く折り返されているだけの継続行(実データで確認: 「決算特別委員会（概況説明、監査報告、...、」+「区役所関係審査）【中継】」のようなケースがある)。この判定ロジックを`domain/committeeMeeting/parseCommitteeCell.ts`の`parseCommitteeCellLines`という純粋関数に切り出し、実データから採取した具体例で回帰テストを書いた(CLAUDE.mdの「テストの方針」に基づく判断)。

### 3. 表示は月ごとの委員会別開催日数のバッジ一覧とする(高度なグラフは作らない)
「年間のマイルストーン」の核心的価値は、予算委員会が例年2〜3月、決算特別委員会が例年9〜10月に集中する、といった年間の周期性が一目でわかることである。これは月ごとに「委員会名(開催日数)」のバッジを並べるだけの単純な一覧で十分に達成できるため、専用のチャートコンポーネントは作らずサイト全体の質素なスタイル(バッジ・カード)に合わせた。

### 4. 取り込みジョブは既存の会期予定表取り込みの直後に実行する
`meetingId`解決のために`Meeting`データが必要なため、cronスケジュールを`scrape-session-schedule`(18:30)の5分後(18:35)に設定した。

## 結果(Consequences)

**メリット**
- 実際にスクレイパーを本番サイトに対して実行し、164件の実データを取得・保存できることを確認した(2025年6月〜2026年7月の範囲、予算委員会が2月・3月、決算特別委員会が9月・10月に集中していることを実データで確認)。
- 回帰テストを追加(`parseCommitteeCell.test.ts`、7件): 単一会議・複数会議・「散会後」の扱い・継続行の連結・空セルの無視・基本名抽出を検証。
- `/milestones`ページで実データが月ごとに正しくグルーピングされ、委員会別の開催日数バッジとして表示されることを確認した。

**デメリット・今後の課題**
- 会議日程一覧ページは直近1年分のみを掲載するローリングウィンドウであるため、複数年分の周期性比較はまだできない(データが蓄積されるにつれて改善される)。
- `meetingId`は、既存の`Meeting`データが会期予定表スクレイピングで限られた件数しか取得されていないため、現時点では164件中65件しか解決できていない(該当する`Meeting`が無い場合はnullのままで、捏造せず正直に「不明」とする設計のため想定通り)。
