# ADR-0004: 認証はOAuth、通知は複数チャネル対応

## ステータス
承認済み(2026-07-06、ユーザーヒアリングにて決定)

## コンテキスト
`users` / `notification_settings` テーブル設計、およびセキュリティ設計(セッション管理、外部連携)に直結する前提だった。

## 検討した選択肢
1. メール+パスワード認証、メール通知のみ
2. OAuth(Google等)ログイン + 複数通知チャネル(採用)
3. 認証機能は将来実装、まずは公開閲覧のみ

## 決定
OAuth(Google等)によるログインを採用し、通知チャネルはメール・LINE・push等の複数チャネルに対応できるスキーマとする。

## 結果(Consequences)

**メリット**
- 自前のパスワード管理・リセットフロー・漏洩リスクを負わない
- ユーザーのログインコストが低く、継続利用の障壁が下がる

**デメリット**
- 外部サービス(Google OAuth、LINE Notify等)への依存が増え、それらのAPI仕様変更の影響を受ける
- 実装は実績あるライブラリ(Auth.js等)を用いて自前実装を避けること(セキュリティ設計 [[../design/01-basic-design.md]] §8)

**将来の拡張性**
- `notification_settings.channel` を列挙型ではなく拡張可能な文字列/JSONで持つことで、新しい通知チャネル追加時にスキーマ変更を不要にする
