# ADR-0009: 非rootの専用デプロイユーザーでプロジェクトを配置・運用する

## ステータス
承認済み(2026-07-06)

## コンテキスト
開発初期に作業ディレクトリを `/root/projects/saitama-council-watch` に作成したが、以下の問題が指摘された。

- `/root` はパーミッション `700` で他ユーザー/ツールから読めず、将来非rootで動かすビルド・デプロイ処理と相性が悪い
- 本番VPSで `root` としてアプリやコンテナ関連ファイルを運用することは、[[0003-hosting-single-vps-docker]] および `docs/design/01-basic-design.md` §8(最小権限原則)と矛盾する

## 決定
- 専用ユーザー `deploy`(非root、パスワードログイン無効・鍵認証のみを本番では想定)を作成
- プロジェクト一式は `/home/deploy/saitama-council-watch` に配置し、所有者を `deploy:deploy` とする
- 本番VPSでも同様に `deploy` ユーザーを作成し、Dockerグループにのみ追加する(sudo権限は付与しない)
- コンテナ内プロセス(Node.js等)もrootではなく非rootユーザーで実行するよう、各Dockerfileで`USER`を明示指定する([[0003-hosting-single-vps-docker]] への追記事項)

## 結果(Consequences)

**メリット**
- ホスト・コンテナ双方で最小権限原則を徹底でき、侵害時の被害範囲を限定できる
- 開発環境と本番環境でユーザー/ディレクトリ構成の慣習を揃えられる

**デメリット**
- ローカル開発時、root操作(ポート80/443のbind等)が必要な場面では都度権限昇格が必要になる(Docker経由の運用では通常問題にならない)

**将来の拡張性**
- 複数人開発になった場合も、`deploy`ユーザーの権限モデルをそのまま拡張(SSH鍵の追加等)すればよく、設計変更は不要
