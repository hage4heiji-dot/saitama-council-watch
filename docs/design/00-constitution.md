# プロジェクト憲法(さいたま市議会ウォッチ)

このドキュメントは、プロジェクト全体を貫く不変の原則を記録するものです。個別の設計判断(ADR)や実装は、必ずこの憲法と矛盾しないことを確認すること。矛盾が生じた場合は、実装ではなくこの憲法との整合性を疑うこと。

## 目的

市民が行政・議会・政治を分かりやすく理解できるWebサービスを作る。単なるニュースサイトではなく、「行政データをAIが整理・可視化するプラットフォーム」を目指す。

## AIの役割と制約(最重要・絶対原則)

Claudeが担当してよい作業は以下のみ:

- 要約
- 分類
- タグ付け
- FAQ生成
- 市民向け説明
- 関連情報抽出

**AIは事実を創作してはいけない。元データが存在しない内容は生成禁止。**

すべてのAI生成コンテンツは、参照元の原本ドキュメント(`documents`テーブル)への外部キーを持たなければならない。原本を持たないAI生成コンテンツは、データベース制約レベルで禁止する([[../adr/0006-ai-content-storage-split]] 参照)。

## 技術スタック(固定)

| レイヤー | 技術 |
|---|---|
| OS | Ubuntu 24.04 LTS |
| Web Server | Nginx |
| Backend | Node.js / Express / TypeScript |
| Frontend | Next.js / TypeScript / TailwindCSS |
| Database(永続データ) | PostgreSQL |
| Database(AI専用) | SQLite |
| ORM | Prisma |
| Version Control | Git |
| AI | Claude Code(開発) / Claude API(実行時) |

### PostgreSQLの役割
永続データ: 議員、会派、会議、議案、条例、投票結果、予算、ユーザー、通知設定。

### SQLiteの役割
AI専用: キャッシュ、Embedding、全文検索、一時データ。**業務データを持たない。**
(公開用AI生成コンテンツの扱いについては [[../adr/0006-ai-content-storage-split]] で修正判断あり)

### 原本データの保存
PDF / HTML / Markdown / JSON は必ず保存する。AIは必ず原本を参照できる構造にする。

## 保守方針

責務を明確に分離する。将来的に以下の変更があっても影響範囲を最小化する設計とする:

- AIモデル変更
- DB変更
- UI変更

## コーディング方針

- SOLID / DRY / KISS / YAGNI を守る
- `any` は禁止
- 例外処理を書く
- 型安全を最優先する

## 開発方針

コードを書く前に、以下の順で設計を完了し、承認を得る:

1. システム全体設計
2. ディレクトリ構成
3. DB設計
4. API設計
5. データフロー
6. AIフロー
7. バッチ設計
8. セキュリティ設計
9. Docker構成
10. 開発ロードマップ

**設計者の承認なしにコードを書かない。**

## この憲法の再実行方法

新しいセッション/エージェントがこのプロジェクトを引き継ぐ場合、以下の順で読むこと:

1. `docs/design/00-constitution.md`(本ファイル) — 不変の制約
2. `docs/adr/*.md` — 各設計判断の背景・検討した代替案・結論
3. `docs/design/01-basic-design.md` — 確定した基本設計(10項目)

これらを読めば、なぜ今の設計になっているかを、ゼロから再現・検証できる。
